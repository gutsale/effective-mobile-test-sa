## Описание
Для реализации отправки PUSH-уведомлений в мобильное приложение предлагается использовать отдельный сервис уведомлений (Notification Service).

PUSH-уведомления могут быть разных типов:
- системные (например, отмена заказа)
- триггерные (например, забытая корзина)
- маркетинговые (рекламные рассылки)

---

## Компоненты системы

- **Mobile App** — мобильное приложение пользователя  
- **API Gateway / BFF** — единая точка входа, маршрутизация и адаптация API под мобильный клиент  
- **Order Service** — сервис управления заказами  
- **Cart Service** — сервис корзины  
- **Marketing Service** — сервис маркетинговых рассылок  
- **Event Bus / Queue** — очередь сообщений (например, Kafka или RabbitMQ)  
- **Notification Service** — сервис отправки уведомлений  
- **Device Token Storage** — хранилище токенов устройств (`user_id → device_token`)  
- **User Notification Settings** — настройки уведомлений пользователя  
- **Notification Templates** — шаблоны уведомлений  
- **Notification DB** — хранение отправленных уведомлений  
- **Scheduler / Rules Engine** — проверка условий и генерация событий по времени  
- **FCM (Firebase Cloud Messaging)** — доставка уведомлений на Android  
- **APNs (Apple Push Notification service)** — доставка уведомлений на iOS  

---

## Общий принцип работы

Мобильное приложение регистрирует device token через API Gateway / BFF, который сохраняется в Device Token Storage.

Сервисы системы генерируют события (например, отмена заказа или неактивная корзина) и отправляют их в Event Bus.

Notification Service подписан на события и выполняет:
- получение события;
- проверку настроек пользователя (разрешены ли уведомления);
- проверку наличия валидного device token;
- проверку допустимого времени отправки;
- получение текста уведомления из Notification Templates;
- формирование сообщения;
- отправку PUSH-уведомления через FCM/APNs;
- сохранение результата в Notification DB.

При временных ошибках используется retry policy с exponential backoff.

## Сценарии

### 1. Отмена заказа

1. В Order Service происходит отмена заказа  
2. Формируется событие `order_cancelled`  
3. Событие отправляется в Event Bus  
4. Notification Service получает событие  
5. Проверяет настройки пользователя и device token  
6. Получает шаблон уведомления  
7. Отправляет PUSH через FCM/APNs  
8. Сохраняет результат  

---

### 2. Забытая корзина

1. Пользователь добавляет товар в корзину  
2. В течение заданного времени не совершает действий  
3. Scheduler определяет неактивную корзину  
4. Формируется событие `abandoned_cart`  
5. Событие отправляется в Event Bus  
6. Notification Service выполняет отправку уведомления  

---

### 3. Маркетинговые уведомления

1. Marketing Service формирует событие рассылки  
2. Событие отправляется в Event Bus  
3. Notification Service получает событие  
4. Выполняет отправку уведомлений пользователям  

---

## Обоснование решения

Использование событийной архитектуры позволяет снизить связанность между сервисами и легко добавлять новые источники уведомлений.

Выделение Notification Service в отдельный сервис позволяет централизовать логику отправки уведомлений и переиспользовать её для разных сценариев.

Использование Scheduler / Rules Engine позволяет реализовать сценарии, зависящие от времени, например неактивную корзину.

Хранение шаблонов уведомлений и настроек пользователей делает систему гибкой и позволяет изменять поведение без изменения кода.

Использование FCM и APNs позволяет доставлять уведомления на мобильные устройства, а retry policy повышает надёжность системы при временных сбоях.
