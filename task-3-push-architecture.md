# Задание 3. Архитектура PUSH-уведомлений

## Описание
Для реализации отправки PUSH-уведомлений в мобильное приложение предлагается использовать отдельный сервис уведомлений (Notification Service).

PUSH-уведомления могут быть разных типов:
- системные (например, отмена заказа)
- триггерные (например, забытая корзина)
- маркетинговые (рекламные рассылки)

---

## Компоненты системы

- **Mobile App** — мобильное приложение пользователя  
- **Backend API** — основной backend системы  
- **Order Service** — сервис управления заказами  
- **Cart Service** — сервис корзины  
- **Marketing Service** — сервис маркетинговых рассылок  
- **Event Bus / Queue** — очередь сообщений (например, Kafka или RabbitMQ)  
- **Notification Service** — сервис отправки уведомлений  
- **Push-провайдеры** — Firebase Cloud Messaging (FCM), Apple Push Notification Service (APNs)  
- **Device Token Storage** — хранилище токенов устройств  

---

## Общий принцип работы

Сервисы системы генерируют события (например, отмена заказа или длительное отсутствие активности в корзине).  
Эти события передаются через очередь сообщений в Notification Service, который принимает решение об отправке PUSH-уведомления.

Notification Service формирует сообщение и отправляет его через push-провайдеров на устройство пользователя.

---

## Сценарии

### 1. Отмена заказа

1. В Order Service происходит отмена заказа  
2. Сервис формирует событие `order_cancelled`  
3. Событие отправляется в очередь сообщений  
4. Notification Service получает событие  
5. Определяет пользователя и его device token  
6. Формирует текст уведомления  
7. Отправляет PUSH через FCM/APNs  
8. Пользователь получает уведомление  

---

### 2. Забытая корзина

1. Пользователь добавляет товар в корзину  
2. В течение заданного времени не совершает действий  
3. Scheduler (планировщик) фиксирует неактивность  
4. Формируется событие `abandoned_cart`  
5. Notification Service отправляет PUSH  

---

### 3. Маркетинговые уведомления

1. Marketing Service формирует рассылку  
2. Передаёт список пользователей  
3. Notification Service отправляет PUSH всем пользователям  

---

## Схема архитектуры

```text
[Mobile App]
     |
     | регистрирует device token
     v
[API Gateway / BFF]
     |
     v
[User/Profile Service] ----> [Device Token Storage]
                                   |
                                   v
                         (user_id → device_token)

[Order Service] --------\
[Cart Service] ----------> [Event Bus / Broker] ---> [Notification Service] ---> [FCM / APNs] ---> [Mobile App]
[Marketing Service] ----/

                          |
                          v
                   [Notification DB]

                          |
                          v
               [Scheduler / Rules Engine]
```

---

## Обоснование архитектуры

Архитура построена с использованием событийного подхода.

Сервисы (Order, Cart, Marketing) генерируют события и отправляют их в очередь (Event Bus), вместо прямой отправки PUSH. Это снижает связанность между сервисами и позволяет обрабатывать события асинхронно.

Для отправки уведомлений выделен отдельный Notification Service. Он централизует логику работы с PUSH, определяет получателей и отправляет сообщения через FCM/APNs.

Device token хранится отдельно, что позволяет адресно отправлять уведомления конкретным пользователям.

Scheduler используется для сценариев, зависящих от времени (например, забытая корзина).
